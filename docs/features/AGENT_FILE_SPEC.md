# AGENT_FILE_SPEC.md

## Overview

`mdcc` is designed as an **agent-first, deterministic document compiler**.
The current output format is a **single PDF**, where charts and tables are rendered as images or formatted HTML tables.

This works well for **human readers**, but it introduces problems for **AI agents** that want to:

* inspect the report
* reuse the underlying data
* perform further analysis
* transform or summarize parts of the document

Agents are significantly better at working with **structured text and structured data** than with rendered images inside PDFs.

However, naïvely exporting the entire document to markdown with all datasets inline introduces a new problem: **context explosion**. Large datasets embedded directly in markdown quickly produce files that are too large for efficient agent processing.

This document describes the **design problem** and proposes a **single-file agent artifact** that preserves mdcc’s core principles.

---

# Core Principles

The design must respect the core philosophy of the mdcc project:

### 1. Single-Source Authoring

The input must remain **a single mdcc document file**.

No multi-file project structures, includes, or external helper files.

---

### 2. Deterministic Execution

Compilation must remain:

* deterministic
* reproducible
* independent of external services
* independent of AI systems

The artifact must not require LLM calls to generate summaries or insights.

---

### 3. Agent-First Design

The system should optimize for:

* machine readability
* deterministic structure
* clear schemas
* easy partial access
* easy transformation and analysis

---

### 4. Single Output Artifact

The compiled artifact must remain **a single file**.

No output folders, no sidecar datasets, no zip bundles.

The artifact should be **portable, self-contained, and immutable**.

---

# The Core Problem

We must reconcile four competing goals:

| Goal                                  | Problem                                |
| ------------------------------------- | -------------------------------------- |
| Charts must preserve underlying data  | Data may contain thousands of rows     |
| Agents prefer textual representations | Raw data in markdown causes file bloat |
| Output must remain one file           | Cannot store sidecar data files        |
| Agents must avoid loading everything  | Large datasets exceed context limits   |

A naïve solution — embedding all chart data into markdown — creates very large documents that agents must load entirely to access even small pieces of information.

The real constraint is not storage.

The real constraint is **bounded access**.

Agents must be able to retrieve **only the subset of information they need**.

---

# Key Insight

The artifact itself **does not need to be fully textual**.

Instead, it must support **efficient selective access** to structured data.

Therefore:

> The agent artifact should be **a single indexed container file** that allows partial retrieval of document components.

---

# Proposed Solution: Agent Artifact Container

Introduce a new compiled output target:

```
mdcc compile report.mdcc --to agent
```

This produces a **single self-contained artifact file**.

Example extension:

```
report.mdca
```

(mdcc agent artifact)

---

# Container Requirements

The container must support:

* single-file storage
* structured data
* indexed queries
* deterministic schema
* stable format

A practical implementation candidate is **SQLite**, which provides:

* single file storage
* built-in indexing
* structured tables
* wide language support
* deterministic behavior
* zero external dependencies

SQLite is used purely as a **container and query engine**, not as a database service.

---

# Artifact Structure

The container stores structured information about the compiled document.

## Document Table

Stores metadata.

Fields:

* `document_id`
* `title`
* `author`
* `date`
* `source_hash`
* `compiler_version`
* `build_timestamp`

---

## Narrative Table

Stores the canonical markdown narrative of the document.

Blocks remain referenced but not expanded with large datasets.

Example structure:

```
# Revenue Analysis

Revenue increased across all regions.

[BLOCK: fig_revenue_trend]

Europe showed the largest growth.

[BLOCK: table_regional_breakdown]
```

---

## Blocks Table

Stores metadata for each executable block.

Fields:

* `block_id`
* `block_type` (`chart`, `table`)
* `caption`
* `label`
* `dataset_id`
* `renderer_type`
* `source_location`

---

## Datasets Table

Metadata about datasets generated by blocks.

Fields:

* `dataset_id`
* `row_count`
* `column_schema`
* `source_block_id`

---

## Data Tables

Each dataset is stored as a dedicated SQL table.

Example:

```
data_fig_revenue_trend
```

Columns correspond to the original dataframe columns.

---

## Profiles Table

Stores **deterministic statistical summaries** for datasets.

These are computed without AI.

Examples:

* row_count
* column_types
* min / max
* mean / median
* null counts
* category cardinality
* quantiles
* first / last values
* sampled preview rows

These summaries allow agents to understand datasets **without loading the full data**.

---

## Artifacts Table

Optional storage for rendered outputs.

Examples:

* chart PNG
* chart SVG
* table HTML
* compiled PDF

This allows the artifact to support both machine and human renderings.

---

# Agent Interaction Model

Agents should not load the entire artifact.

Instead, they interact through the mdcc CLI.

Examples:

```
mdcc inspect report.mdca
```

List document metadata.

---

```
mdcc sections report.mdca
```

List narrative sections.

---

```
mdcc show report.mdca --section Results
```

Render a specific section as markdown.

---

```
mdcc block show report.mdca --id fig_revenue_trend
```

Return block metadata.

---

```
mdcc data head report.mdca --block fig_revenue_trend --rows 20
```

Retrieve preview rows.

---

```
mdcc data query report.mdca --block fig_revenue_trend --sql "select date, revenue from data_fig_revenue_trend limit 50"
```

Run bounded queries.

---

```
mdcc render report.mdca --section Results --to markdown
```

Generate markdown for a specific section.

# Why This Design Works

This architecture satisfies all constraints:

| Requirement              | Result    |
| ------------------------ | --------- |
| Single source file       | preserved |
| Single compiled artifact | preserved |
| Deterministic execution  | preserved |
| No AI dependency         | preserved |
| Efficient agent usage    | enabled   |
| Large datasets           | supported |
| Partial retrieval        | supported |

The artifact remains **portable, deterministic, and self-contained**.

Agents interact with it through **bounded queries and targeted rendering**.

---

# Deterministic Chart Semantics

Charts must not rely on LLM summaries.

Instead, deterministic metadata is stored:

Example for a line chart:

* chart_type
* x_field
* y_field
* series_fields
* dataset_row_count
* x_domain
* y_min / y_max
* first_value
* last_value
* quantiles
* sample_rows

Agents can derive further insights themselves.

---

# Why Markdown Alone Is Not Enough

Pure markdown exports fail for large data because they:

* embed entire datasets inline
* produce extremely large files
* require agents to load everything
* waste context window capacity

The container model solves this by separating:

* **storage**
* **retrieval**
* **rendering**

# Future Extensions

Potential future improvements:

### Dataset Sampling

Store deterministic downsampled previews for very large datasets.

---

### Block-Level Rendering

Allow exporting individual blocks as standalone markdown snippets.

---

### External Analysis

Allow exporting datasets directly:

```
mdcc export dataset report.mdca fig_revenue_trend --to csv
```

---

### Optional Human Bundle

Embed both:

* `report.pdf`
* agent container

into the same artifact.

---

# Summary

The correct agent-first design is **not a giant markdown file**.

Instead, mdcc should compile to a **single self-contained container artifact** that:

* stores the document structure
* stores datasets
* stores deterministic profiles
* allows efficient selective access

Agents interact with the artifact through **bounded CLI queries and rendering commands**.

This preserves mdcc’s core design principles while enabling large-scale agent-driven analysis.
